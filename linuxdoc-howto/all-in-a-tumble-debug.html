kernel-doc from all-in-a-tumble.[ch]¶

Below you find the reST markup, generated from kernel-doc comments of the example files source all-in-a-tumble.h and source all-in-a-tumble.c. This markup will be produced by the kernel-doc parser and inserted in the document by using the following directives:

.. kernel-doc::  ./all-in-a-tumble.c
   :module: foo

.. kernel-doc::  ./all-in-a-tumble.h
   :module: foo

The option :module: is optional, to find out why we use this option here, see kernel-doc options.

427.. _`foo.rst_mode.highlighting`:
428
429highlighting
430------------
431
432The highlight pattern, are non regular reST markups. They are only available
433within kernel-doc comments, helping C developers to write short and compact
434documentation.
435
436- \ :c:func:`user_function`\  : function
437- \ ``a``\  : name of a parameter
438- \ :c:type:`struct my_struct <my_struct>`\  : name of a structure (including the word struct)
439- \ :c:type:`union my_union <my_union>`\  : name of a union
440- \ :c:type:`my_struct->a <my_struct>`\  or \ :c:type:`my_struct.b <my_struct>`\  -  member of a struct or union.
441- \ :c:type:`enum my_enum <my_enum>`\  : name of a enum
442- \ :c:type:`typedef my_typedef <my_typedef>`\  : name of a typedef
443- \ ``CONST``\  : name of a constant.
444- \ ``$ENVVAR``\  : environmental variable
445
446The kernel-doc parser translates the pattern above to the corresponding reST
447markups. You don't have to use the *highlight* pattern, if you prefer *pure*
448reST, use the reST markup.
449
450- :c:func:`user_function` : function
451- ``a`` : name of a parameter
452- :c:type:`struct my_struct <my_struct>` : name of a structure (including the word struct)
453- :c:type:`union my_union <my_union>` : name of a union
454- :c:type:`my_struct->a <my_struct>` or :c:type:`my_struct.b <my_struct>` -  member of a struct or union.
455- :c:type:`enum my_enum <my_enum>` : name of a enum
456- :c:type:`typedef my_typedef <my_typedef>` : name of a typedef
457- ``CONST`` : name of a constant.
458- ``$ENVVAR`` : environmental variable
459
460Since the prefixes ``$...``, ``&...`` and ``@...`` are used to markup the
461highlight pattern, you have to escape them in other uses: $lorem, &lorem,
462%lorem and @lorem. To esacpe from function highlighting, use lorem().
463

619.. _`foo.vintage.parser-mode`:
620
621Parser Mode
622-----------
623
624\*vintage\* kernel-doc mode
625
626Within the \*vintage kernel-doc mode\* ignores any whitespace or inline
627markup.
628
629- Inline markup like \*emphasis\* or \*\*emphasis strong\*\*
630- Literals and/or block indent:
631
632a + b
633
634In kernel-doc \*vintage\* mode, there are no special block or inline markups
635available. Markups like the one above result in ambiguous reST markup which
636could produce error messages in the subsequently sphinx-build
637process. Unexpected outputs are mostly the result.

932Description
933-----------
934
935Write a value to the iosys_map considering its layout is described by a C
